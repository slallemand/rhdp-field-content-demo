= Module 2: Provisionner avec OpenShift Virtualization (CNV)
:imagesdir: ../assets/images

== Introduction

*Durée: ~10 minutes*

Ce module vous guide à travers le processus de provisionnement d'un cluster OpenShift avec OpenShift Virtualization (CNV) et le déploiement de votre Field Content via GitOps.

== Objectifs d'apprentissage

À la fin de ce module, vous serez capable de:

* Commander un item "Field Content CI" depuis le catalogue RHDP
* Comprendre le rôle de CNV dans le provisionnement rapide
* Explorer la structure d'un projet Field Content
* Visualiser le déploiement GitOps dans ArgoCD UI
* Tester le flux GitOps end-to-end

== Points clés à démontrer

[.tile]
.Know
****
*Messages clés de ce module:*

. *Provisionnez rapidement* - Clusters OpenShift multi-cloud en minutes avec CNV
. *Automatisez le déploiement* - Field Source Content déploie via GitOps (Ansible/Helm)

OpenShift Virtualization (CNV) transforme le provisionnement d'un cluster de plusieurs heures à quelques minutes. Combiné avec GitOps, vous obtenez un environnement de démo entièrement automatisé de bout en bout.
****

== Partie 2: Explorer le Field Content déployé

=== Étape 5: Structure du repository

[.tile]
.Show
****
Ouvrez le repository link:https://github.com/slallemand/rhdp-field-content-demo[rhdp-field-content-demo,window=_blank] dans un nouvel onglet.

Explorez la structure:

[source,text]
----
rhdp-field-content-demo/
├── Chart.yaml              # Helm chart principal (root)
├── values.yaml             # Valeurs par défaut + RHDP injectées
├── templates/              # Templates ArgoCD Applications
│   └── application.yaml    # Crée l'app "simple-app"
├── components/             # Charts de composants individuels
│   └── simple-app/
│       ├── Chart.yaml
│       ├── values.yaml
│       └── templates/
│           ├── deployment.yaml
│           ├── service.yaml
│           └── route.yaml
└── showroom-content/       # Contenu de documentation (ce guide!)
    ├── site.yml
    ├── antora.yml
    └── content/
----
****

[.tile]
.Know
****
*Pattern App of Apps en action:*

. *Niveau 1 (root):* Chart Helm principal
  ** Déployé automatiquement par RHDP via ArgoCD
  ** Crée des ArgoCD Applications enfants

. *Niveau 2 (templates/):* ArgoCD Applications
  ** Chaque fichier dans `templates/` = une Application ArgoCD
  ** Pointe vers un composant dans `components/`

. *Niveau 3 (components/):* Charts de composants
  ** Chaque sous-répertoire = un composant indépendant
  ** Déployé par son ArgoCD Application parente

*Avantages:*

* Modularité: composants déployables indépendamment
* Réutilisabilité: un composant peut être référencé par plusieurs démos
* Flexibilité: ajoutez/supprimez des composants sans tout reconstruire
****

=== Étape 6: Examiner le Chart principal

[.tile]
.Show
****
Ouvrez `Chart.yaml` à la racine:

[source,yaml]
----
apiVersion: v2
name: field-content
description: RHDP Field Content Demo
type: application
version: 1.0.0
appVersion: "1.0"
----

Ouvrez `values.yaml`:

[source,yaml]
----
deployer:
  apiUrl: ""      # ← Injecté par RHDP
  domain: ""      # ← Injecté par RHDP

repository:
  url: https://github.com/slallemand/rhdp-field-content-demo
  ref: main
----
****

[.tile]
.Know
****
*Valeurs injectées automatiquement par RHDP:*

[WARNING]
====
Les valeurs `deployer.apiUrl` et `deployer.domain` sont vides dans Git, car elles sont injectées dynamiquement par RHDP au moment du déploiement.

Exemples de valeurs injectées:

* `deployer.apiUrl`: `https://api.cluster-abc123.example.com:6443`
* `deployer.domain`: `apps.cluster-abc123.example.com`

Ces valeurs changent pour chaque cluster provisionné - ne les hard-codez JAMAIS dans Git!
====

*Comment RHDP injecte ces valeurs:*

Le workload `ocp4_workload_field_content` crée l'ArgoCD Application avec:

[source,yaml]
----
spec:
  source:
    helm:
      values: |
        deployer:
          apiUrl: {{ ocp_api_url }}
          domain: {{ ocp_apps_domain }}
----

Ansible résout ces variables à partir de l'inventaire du cluster provisionné.
****

=== Étape 7: Examiner l'ArgoCD Application template

[.tile]
.Show
****
Ouvrez `templates/application.yaml`:

[source,yaml]
----
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: simple-app
  namespace: openshift-gitops
  labels:
    demo.redhat.com/application: "field-content"  # ← Label OBLIGATOIRE
spec:
  project: default
  source:
    repoURL: {{ .Values.repository.url }}
    targetRevision: {{ .Values.repository.ref }}
    path: components/simple-app                   # ← Pointe vers le composant
    helm:
      valueFiles:
        - values.yaml
  destination:
    server: {{ .Values.deployer.apiUrl }}         # ← Valeur RHDP injectée
    namespace: simple-app
  syncPolicy:
    automated:
      prune: true                                  # ← Supprime ressources obsolètes
      selfHeal: true                               # ← Corrige les drifts
    syncOptions:
      - CreateNamespace=true                       # ← Crée le namespace automatiquement
----
****

[.tile]
.Know
****
*Décortiquons les éléments clés:*

[cols="1,3"]
|===
|Élément |Explication

|`labels.demo.redhat.com/application`
|*OBLIGATOIRE* pour monitoring RHDP et cleanup automatique

|`source.repoURL`
|Templated avec `{{ .Values.repository.url }}` pour réutilisabilité

|`source.path`
|Pointe vers `components/simple-app` - le sous-chart à déployer

|`destination.server`
|Utilise `{{ .Values.deployer.apiUrl }}` injecté par RHDP

|`syncPolicy.automated.prune`
|Supprime automatiquement les ressources supprimées de Git

|`syncPolicy.automated.selfHeal`
|Restaure l'état Git si modification manuelle (`oc edit`)

|`syncOptions.CreateNamespace`
|ArgoCD crée le namespace s'il n'existe pas
|===

[IMPORTANT]
====
Le label `demo.redhat.com/application: "field-content"` sert à:

. *Health monitoring:* RHDP surveille les Applications avec ce label
. *Auto-cleanup:* À la fin de vie du cluster, RHDP supprime toutes les ressources labellisées
. *Metrics:* Statistiques de déploiement et d'utilisation
====
****

=== Étape 8: Examiner un composant

[.tile]
.Show
****
Explorez `components/simple-app/`:

[source,text]
----
components/simple-app/
├── Chart.yaml
├── values.yaml
└── templates/
    ├── deployment.yaml
    ├── service.yaml
    └── route.yaml
----

Ouvrez `components/simple-app/templates/route.yaml`:

[source,yaml]
----
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: simple-app
  namespace: simple-app
spec:
  host: simple-app.{{ .Values.deployer.domain }}  # ← Domaine dynamique
  to:
    kind: Service
    name: simple-app
  port:
    targetPort: http
  tls:
    termination: edge
    insecureEdgeTerminationPolicy: Redirect
----
****

[.tile]
.Know
****
Notez comment `{{ .Values.deployer.domain }}` est utilisé pour générer automatiquement l'URL de la Route.

*Résultat après rendering:*

Si `deployer.domain = apps.cluster-abc123.example.com`, la Route sera:

[source,yaml]
----
spec:
  host: simple-app.apps.cluster-abc123.example.com
----

*Pourquoi c'est génial:*

. Même code fonctionne sur *tous* les clusters RHDP
. Aucun hard-coding d'URLs
. Pas de configuration manuelle post-déploiement
. Reproductibilité parfaite

[TIP]
====
Vous pouvez transmettre `deployer.domain` à n'importe quel niveau de la hiérarchie Helm via `values.yaml` ou `helm.parameters` dans l'ArgoCD Application.
====
****

== Partie 4: Tester le flux GitOps

=== Étape 12: Modifier le code et observer la synchronisation

[.tile]
.Show
****
Démontrons le flux GitOps complet en temps réel.

*Option A: Via GitHub UI (plus simple pour la démo)*

. Ouvrez link:https://github.com/slallemand/rhdp-field-content-demo/blob/main/components/simple-app/values.yaml[simple-app/values.yaml,window=_blank] dans GitHub
. Cliquez sur l'icône *Edit* (crayon)
. Changez `replicaCount: 1` → `replicaCount: 2`
. Scrollez en bas, ajoutez un commit message: "Scale simple-app to 2 replicas"
. Cliquez sur *Commit changes*

*Option B: Via Git CLI (pour les SA qui préfèrent le terminal)*

[source,bash,role="execute"]
----
# Cloner le repo (ou fork si lecture seule)
git clone https://github.com/slallemand/rhdp-field-content-demo.git
cd rhdp-field-content-demo

# Modifier le fichier
vi components/simple-app/values.yaml
# Changer replicaCount: 1 → replicaCount: 2

# Committer et pusher
git add components/simple-app/values.yaml
git commit -m "Scale simple-app to 2 replicas"
git push origin main
----
****

[.tile]
.Know
****
*Ce qui se passe après le Git push:*

[cols="1,3,1"]
|===
|Temps |Action |Acteur

|t+0s
|Commit pushé sur GitHub
|Vous

|t+0-180s
|ArgoCD poll le repository (intervalle par défaut: 3 min)
|ArgoCD

|t+180s
|ArgoCD détecte le changement dans `components/simple-app/values.yaml`
|ArgoCD

|t+181s
|ArgoCD compare l'état Git vs l'état cluster (OutOfSync détecté)
|ArgoCD

|t+182s
|ArgoCD lance la synchronisation automatique (`syncPolicy.automated`)
|ArgoCD

|t+183s
|Helm chart re-rendered avec `replicaCount: 2`
|ArgoCD (Helm)

|t+184s
|Deployment `simple-app` mis à jour: `spec.replicas: 2`
|ArgoCD (kubectl apply)

|t+185s
|Kubernetes crée un deuxième Pod
|Kubernetes

|t+200s
|Pod #2 passe à *Running*, Application passe à *Healthy + Synced*
|Kubernetes + ArgoCD
|===

[NOTE]
====
Vous pouvez réduire l'intervalle de polling d'ArgoCD de 3 minutes à 1 minute ou moins, mais cela augmente la charge sur le cluster. Pour une démo, 3 minutes est acceptable.

Ou utilisez *ArgoCD Refresh* pour forcer la synchronisation immédiate (bouton dans l'UI).
====
****

=== Étape 14: Démontrer le self-healing

[.tile]
.Show
****
Démontrons maintenant le self-healing d'ArgoCD.

*Dans la console OpenShift:*

. Allez dans *Workloads* → *Deployments*
. Sélectionnez le projet *simple-app*
. Cliquez sur le Deployment *simple-app*
. Cliquez sur *YAML* (onglet en haut)
. Trouvez `spec.replicas: 2` et modifiez à `spec.replicas: 3`
. Cliquez sur *Save*

*Dans ArgoCD UI:*

. Observez l'Application *simple-app*
. Après quelques secondes (max 30s), vous verrez:
  ** Statut passe brièvement à *OutOfSync*
  ** ArgoCD détecte le drift (Git dit 2, cluster a 3)
  ** Avec `selfHeal: true`, ArgoCD restaure automatiquement à 2 replicas
  ** Statut retourne à *Synced* + *Healthy*

*Résultat:* Le 3ème Pod est supprimé automatiquement, retour à 2 replicas.
****

[.tile]
.Know
****
*Pourquoi le self-healing est critique:*

[cols="1,2,2"]
|===
|Scénario |Sans self-heal |Avec self-heal

|SA modifie en live pour debug
|Drift non détecté, incohérence
|Drift détecté et corrigé auto

|Client demande un changement urgent
|`oc edit`, oubli de committer
|Forcé de passer par Git

|Erreur humaine (`oc delete`)
|Ressource supprimée définitivement
|Ressource recréée automatiquement

|Script automatisé bugué
|Modifications anarchiques
|Restauration continue de l'état Git
|===

*Message clé:* Git est la source de vérité unique. Toute modification en dehors de Git est temporaire.

[IMPORTANT]
====
Dans un environnement de production, `selfHeal` peut être trop agressif (empêche le debug en live). Pour RHDP, c'est idéal car:

. Environnements éphémères (durée de vie courte)
. Objectif = démo reproductible
. Évite les drifts accidentels avant une démo client
====
****

== Points de discussion avancés

[.tile]
.Know
****
*Pour les participants expérimentés:*

. *Branching strategy:* Comment gérer dev/staging/prod avec GitOps?
  ** *Approche:* Branches Git (`dev`, `staging`, `main`) + ArgoCD Applications par environnement

. *Secrets management:* Comment gérer les secrets sans les committer?
  ** *Solutions:* Sealed Secrets, External Secrets Operator, Vault

. *Multi-cluster:* Comment déployer la même démo sur plusieurs clusters?
  ** *Approche:* ArgoCD ApplicationSet avec générateurs (git, cluster, list)

. *Rollback strategy:* Que faire si une sync introduit un bug?
  ** *Solution rapide:* `git revert` + push
  ** *Solution chirurgicale:* ArgoCD UI → History → Rollback to previous revision

. *Monitoring:* Comment surveiller la santé de mes démos déployées?
  ** *RHDP:* Health checks automatiques via le label `demo.redhat.com/application`
  ** *Custom:* Prometheus + Grafana, alertes sur statut ArgoCD
