* Comprendre le rôle du Skills Marketplace

[.tile]

*Qu'est-ce qu'une Skill?*

Une skill est un composant réutilisable qui peut:

[cols="1,3"]
|===
|Type |Exemples

|*Installation automatisée*
|Déployer Kafka, PostgreSQL, AMQ Streams

|*Configuration*
|Configurer des pipelines Tekton, créer des secrets

|*Données de démo*
|Charger des datasets, créer des utilisateurs test

|*Monitoring*
|Installer Grafana dashboards, alertes Prometheus

|*Nettoyage*
|Scripts de teardown, suppression de ressources
|===

Chaque skill est un rôle Ansible ou un Helm chart prêt à l'emploi.

=== Étape 3: Intégrer une skill dans votre démo

[.tile]
.Show
****
Pour utiliser une skill dans votre Field Content, deux approches:

*Approche 1: Rôle Ansible (via ocp4_workload_field_content)*

Ajoutez dans votre `agnosticd` config:

[source,yaml]
----
ocp4_workload_field_content_workloads:
  - name: install-kafka
    type: ansible
    git_url: https://github.com/rhpds/rhdp-skills-marketplace.git
    git_path: skills/install-kafka
    vars:
      kafka_namespace: kafka-demo
      kafka_replicas: 3
----

*Approche 2: Helm chart (via ArgoCD Application)*

Créez une ArgoCD Application dans `templates/`:

[source,yaml]
----
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: kafka-skill
  labels:
    demo.redhat.com/application: "field-content"
spec:
  source:
    repoURL: https://github.com/rhpds/rhdp-skills-marketplace
    path: skills/install-kafka-helm
    targetRevision: main
  destination:
    server: {{ .Values.deployer.apiUrl }}
    namespace: kafka-demo
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
----
****

.Know
****
*Avantages d'utiliser les skills:*

. *Gain de temps* - Ne réécrivez pas ce qui existe déjà
. *Qualité testée* - Skills validées par la communauté
. *Maintenance partagée* - Mises à jour centralisées
. *Documentation* - README standardisé pour chaque skill
. *Évolutivité* - Ajoutez facilement des fonctionnalités à vos démos

== Bonnes pratiques de création de contenu

=== Étape 4: Standards de qualité

[.tile]
.Know
****
*Checklist de qualité pour vos Field Content:*

[cols="1,3"]
|===
|Catégorie |Bonnes pratiques

|*Structure*
a|* Utilisez la structure template officielle
* Séparez les composants dans `components/`
* Helm charts au lieu de YAML statique

|*GitOps*
a|* Toutes les ArgoCD Apps ont le label `demo.redhat.com/application: "field-content"`
* Utilisez `syncPolicy.automated` pour le sync automatique
* `CreateNamespace=true` sur toutes les Applications

|*Variables*
a|* Utilisez `{{ .Values.deployer.apiUrl }}` et `{{ .Values.deployer.domain }}`
* Ne hard-codez jamais des URLs ou domaines
* Documentez toutes les variables dans `values.yaml`

|*Tests*
a|* Exécutez `helm lint` avant chaque commit
* Testez avec `helm template` en local
* Validez sur un cluster de dev avant production

|*Documentation*
a|* README.md à la racine expliquant le contenu
* Showroom content pour les labs interactifs
* Commentaires dans les templates Helm
|===
****

=== Étape 6: Contribuer vos skills

[.tile]
.Show
****
Pour partager une skill avec la communauté RHDP:

. Créez votre skill localement avec la structure standard
. Documentez dans un `README.md`:
  ** Description de la skill
  ** Paramètres disponibles
  ** Exemples d'utilisation
  ** Prérequis
. Testez sur un cluster RHDP
. Ouvrez une Pull Request sur link:https://github.com/rhpds/rhdp-skills-marketplace[rhdp-skills-marketplace,window=_blank]
. L'équipe RHDP review et merge
****

[.tile]
.Know
****

[cols="1,2"]
|===
|Raison |Bénéfice

|*Visibilité*
|Votre travail profite à toute la communauté Red Hat

|*Maintenance partagée*
|D'autres contributeurs améliorent votre skill

|*Standardisation*
|Tous utilisent les mêmes patterns éprouvés

|*Reconnaissance*
|Crédit pour vos contributions (CONTRIBUTING.md)
|===

Les skills les plus populaires sont souvent mises en avant dans les communications RHDP et peuvent devenir des composants officiels du catalogue.
